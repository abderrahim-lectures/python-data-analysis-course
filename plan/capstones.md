# Real-World Projects: an open, ongoing series (formerly "Capstone Projects")

An open library of practical, install-Python-for-real projects — the natural graduation step off the in-browser JupyterLite playground the rest of the course relies on. Originally shipped as "Capstone Projects" (one new, year-labeled project per year, gated behind finishing the whole course); redesigned into an ongoing series so new projects can be added whenever a trending tool or framework is worth a real, hands-on write-up — not on a fixed yearly schedule, and not gated behind course completion.

- **No completion gate.** `CapstoneChooser` (`src/components/CapstoneChooser`) lists every project, freely browsable and linkable at any time — including from the homepage. Each project's own intro states what background it expects (usually Python 101, sometimes Data Analysis too) instead of relying on a hard unlock condition. `useCourseComplete()` (`src/hooks/useUnlockCondition.ts`) still exists, but only drives the congratulations banner on `progress.tsx` — it no longer gates any content.
- **Content structure — one folder per project, no year prefix in the visible title:**
  ```
  docs/projects/
  ├── index.mdx                  # "Real-World Projects" — <CapstoneChooser> lists every project, newest first
  ├── ai-agent/
  │   ├── _category_.json
  │   └── index.md                # install Python for real, build an AI agent with deepagents
  └── finetune-llm-unsloth/
      ├── _category_.json
      └── index.md                # fine-tune a small LM with Unsloth
  ```
  `sidebars.ts` has `projectsSidebar` (`dirName: 'projects'`), matching `python101Sidebar`/`dataAnalysisSidebar` — the navbar's "Projects" item is a `docSidebar` link like the other two sections.
- **Structural metadata lives once, not per-locale:** `src/data/projects.ts` exports `PROJECTS: {id, date, url, tags}[]` — the single source of truth for each project's non-translated fields. `date` (ISO, e.g. `"2027-07"`) drives `CapstoneChooser`'s newest-first sort and is never rendered. `docs/projects/index.mdx` (and its 3 i18n copies) import `mergeProjectMeta()` to merge this shared metadata with each locale's own literal `title`/`summary` strings — Docusaurus's static i18n extraction needs non-templated `<Translate>` ids, so translated text still has to be written per-locale, but id/date/url/tags no longer are. The homepage does the equivalent by hand, importing `PROJECTS` directly and writing one static `<Translate>` block per project (same idiom as the homepage's two section cards) — adding a project means one new entry in `src/data/projects.ts` plus one new block in `src/pages/index.tsx` and `docs/projects/index.mdx` (×4 locales) for the translated text.
- **Tags:** each project has a `tags: string[]` (e.g. `["Fine-tuning", "LoRA", "LLMs"]`), rendered as small pills on its card — both on `/docs/projects` and the homepage. No filtering UI yet, just display. `"Community"` is a valid tag for a future learner-contributed project, shipped the same way as any other project (a plain PR to `docs/projects/<slug>/`) — no separate submission backend.
- **Per-project completion tracking — unchanged:** each project page ends with `<CapstoneProgressCheckbox capstoneId="..." />` (`src/components/CapstoneProgressCheckbox`, mirroring `ProgressCheckbox`'s week-level pattern) — checking it writes `pda-course:capstone-progress[capstoneId] = true` and awards a `capstone-<id>-complete` badge (`capstoneCompleteBadge()` in `src/utils/badges.ts`; `describeBadge.ts` renders it as "Project complete: …").
- **Backward compatibility — id strings are permanent, everything else can change:** `"2026-ai-agent"` and `"2027-finetune-llm"` are the exact ids already stored in real students' `localStorage` (`pda-course:capstone-progress` map, `capstone-<id>-complete` badges) from before this redesign. These ids stay unchanged forever even though their folder names, URLs (`/docs/bonus/2026-ai-agent` → `/docs/projects/ai-agent`), and displayed titles (year prefix dropped) all changed — the id is purely an internal key, decoupled from anything user-visible. `@docusaurus/plugin-client-redirects` (configured in `docusaurus.config.ts`) redirects the old `/docs/bonus` URLs to their new `/docs/projects` equivalents.
- **Maintenance policy — frozen except critical fixes:** once a project ships, its content stays as originally written, but broken links, dead/deprecated APIs, and security issues still get patched via the normal issue→branch→PR flow. A newer, better approach to the same theme is a new project, not a rewrite of an old one.
- **Runnable example code, one folder per project:** `examples/<slug>/` (e.g. `examples/ai-agent/`, `examples/finetune-llm-unsloth/`) — each genuinely specific to its project's theme, no generic "starter template" to keep in sync. `examples/student-projects/` is the PR-based submission gallery for real-world projects students have built themselves (renamed from `examples/student-agents/` since it's no longer agent-specific).
- **Why this design fits here:** it's a genuine motivator (a taste of applied, install-it-for-real work) that pairs naturally with finishing a section's fundamentals, and it's a clean place to introduce local installs/environment variables/API keys — topics that would be premature in Weeks 1–10 given the "students can't install Python yet" constraint the whole playground architecture is built around. Making it an open, ongoing series rather than a once-a-year gated reward means the course accumulates value continuously, and removing the completion gate means the projects double as a discoverability/marketing surface for visitors who land on the homepage before ever starting the course.
